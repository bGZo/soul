        - ## 组策略及其优势
          collapsed:: true
          - 咱们先来谈谈被各种「优化」、「加速」、「隐私」相关的技巧频繁点名的组策略。
          - 抛开这些影响因子多且难以量化的概念，组策略里确实有一些好用乃至必要的选项：比方说 Windows 和厂商推送的显卡驱动版本常常落后于显卡厂商，对于游戏玩家而言，后者的性能往往更好，此时我们需要用组策略关闭 Windows 驱动更新；再比方说，经常使用远程桌面的用户可以调整组策略中的非常详细的远程桌面设置，提升连接体验……
          - 但它并不是什么旁门左道。策略自然指的是某种规划、政策，但组策略名字却有些让人摸不着头脑，一个人用着一台电脑，哪来的「组」呢？如果你有这个疑问，可以在 shell 中执行 `gpresult.exe /R`，这是一个组策略审计程序，它会给出如下输出：
          - ![](https://cdn.sspai.com/2023/03/20/article/feaa89718bd3512daa1b8379516d0cfb)
          - 可以看出，组指的是安全组（security groups），并且本地组策略只是组策略的一种，这些概念来自 Windows 域网络环境。在一个公司中，IT 管理员通过 Windows Server 管理成百上千台电脑，这些电脑被分成不同的组：
          - ![](image ss-img-wrapper)
          - 安全组 | Microsoft
          - 管理员想要为一部分电脑统一安装某个软件，或者统一屏蔽某个网站，需要为该组创建一些特定的策略，我们将其称之为组策略对象（**Group Policy Object**，简称 GPO），然后借助公司网络将策略应用到这个安全组：
          - ![](image ss-img-wrapper)
          - 特定组的策略 | Microsoft
          - Windows 非家庭版的用户搜索 `gpedit.msc` 打开组策略编辑器，能发现 `软件设置` 目录是空的，这也是因为该策略其实是用于远程服务器统一推送安装软件的。普通用户电脑上的和位于服务器上的组策略相对应，才会被称为「本地」组策略，在英文资料更常见其缩写：**LGPO（Local Group Policy Object）**。
          - ![](image ss-img-wrapper)
          - 在企业网络中，当公司不允许员工访问某个网站，更改某些功能时，显然员工理应不能通过任何正常手段绕过这些限制。对于普通用户而言，这意味着我们手上的本地组策略编辑器也有相同的功能：执行强制且彻底的应用策略。
          - 强制意味着无论系统或应用原先的设置如何，它们都会被调整为组策略指定的状态，并且用户、应用本身不能以任何正常手段修改这些设置。例如，设置 Chrome 不再检查是否为默认浏览器后，该选项会消失，替代以「Google Chrome cannot determine or set the default browser」的提示：

          - 和系统及应用内的部分选项不同，组策略提供的设置往往是彻彻底底的开或关。譬如在 Windows 设置 > 任务栏设置中关闭 Teams 和小组件的选项后，这两个应用的进程仍然留存在后台，用户仍然可以通过全局快捷键 `⊞Win + W` 和 `⊞Win + C` 的方式调出它们的主界面。但用策略的方式修改后，Teams 选项会消失，即使没有用 `winget` 的方法卸载小组件包，其进程也不会启动，全局快捷键也没有作用。

          - 除了以上两个特点，由于企业环境的特殊性，开发商会在组策略中提供一些并没有暴露给一般用户的选项。这意味着你能在组策略中找到更改 Windows 系统更新、控制应用程序遥测、禁用 Firefox Pocket 等实用开关。

        - ## 使用组策略管理模板
          collapsed:: true
          - 因为以上三个特点，组策略编辑器成为了调教 Windows 的神器之一。而调教的方法便是使用组策略编辑器中的「管理模板」目录。虽然 LGPO 在 Windows 设置目录下中提供了其它策略，但其中大部分对普通用户来说用处不大，因而本文不做讨论。
          - ### 前提
            collapsed:: true
            - 组策略编辑器仅在**非** Windows 家庭版本中可用，家庭版用户如果不想升级到专业版，也能通过一些方法用到组策略编辑器，例如：
              - * 使用 dism 添加组策略管理单元：下载 [此脚本](https://github.com/GDaily/gpedit-msc/blob/maste/gpedit-msc.bat)，以管理员权限执行，等待脚本执行完成；
              - * 使用第三方组策略编辑器：如 [PolicyPlus](https://github.com/2FFleex255/PolicyPlus)，功能和官方的大致相同。
            - 大多数管理模板都对应着某个注册表项，因而如果不愿意安装、使用组策略编辑器，亦可以结合本节内容，参看下一节定位、修改注册表的方案。
            - 但无论是修改组策略还是注册表都需要管理员权限。此外，如果你的电脑受公司组策略统一管理，本地组策略的可能不会生效，你只能修改公司没有配置的策略。
          - ### 打开、查找管理模板
            collapsed:: true
            - 在开始菜单搜索「组策略」或者 `gpedit.msc`，点击便可打开组策略编辑器。
            - `gpedit.msc` 的后缀名并非 exe 而是 msc ，这是因为组策略编辑器是作为 Windows 管理控制台（MMC，应用名 mmc.exe）的插件而存在的，微软称之为管理单元（snap-in），后缀 msc 代表的正是 Microsoft Windows management console。除了它之外，Windows 上的服务、任务计划程序等也都是 MMC 的管理单元。==实际上，你也可以先打开 mmc.exe，然后按 `⌃Ctrl + M` 找到 LGPO，在那里它叫组策略对象编辑器。==
            -
            - 如上图所示，本地组策略默认提供了两个分类：计算机配置和用户配置，前者对计算机所有用户生效，后者只对当前用户生效。除非有特别需求，都建议修改计算机配置中的管理模板，用户配置可能存在权限不足的问题。如果同时指定了计算机配置和用户配置，那么计算机配置优先级更高。
            -
            - 点击并展开计算机配置中的管理模板，目录结构中包含了所有可供配置的策略了。你当然可以按目录一个个翻阅，不过更高效的方法是利用组策略编辑器提供的筛选功能。

            - 最后点击确定，或者右键选择「启用筛选器」。由于我想找的策略和 Windows 组件有关，所以我直接点击管理模板下的「Windows 组件」目录，最终就找到了「允许小组件」策略：
            - Windows 自带的中文翻译一向质量不高，有需求的读者可以将系统调整为英文后检索，效率更高一些。
            - 除了自带的筛选功能，在线网站 [Group Policy Search](https://gpsearch.azurewebsites.net) 提供了更加全面友好的组策略搜索体验。访问网站，输入关键词，辅以通配符，便能得到策略的详细位置和具体信息。这个网站之后我们还会用到。

          - ### 配置管理模板
            collapsed:: true
            - 找到了想要的条目，就能根据编辑器提供的说明配置组策略。
            - 所有策略都有三种状态：未配置，已启用和已禁用，已启用状态下部分策略会要求具体指定一些数值。具体操作在参考策略的帮助页面有非常详细的描述，在此不做展开：
            - ![](image ss-img-wrapper)
            - Windows 自动更新策略配置界面，可以配置手动下载更新
            - 唯一需要注意的是某些策略的限制，这些限制大概分为三类：
              - * 平台、系统版本限制：如「默认启用通过质量更新推出的功能」要求系统版本为 Windows 11 22H2；
              - * 应用版本限制：如「禁用 Pocket」要求 Firefox 版本号大于 60；
              - * 依存关系：Google Chrome 退出清理浏览器数据要求关闭数据同步；Edge「允许安装」策略要求 Windows 加入域网络；
            - 尤其留意最后一种限制，它们分布在不同策略的说明部分，很容易被忽略。另外，还有少数策略，例如禁用运行（`⊞Win + R`），虽然写着不支持新版 Windows，实际上也是有效果的。组策略修改还原都比较容易，试试有无效果也未尝不可。
            - 部分策略修改后会立即生效，也有一些需要重启应用和系统才会完全生效。但如果两次重启后策略仍然看不到效果，应当检查组策略是否配置正确，或者是否有第三方软件更改了同样的策略。

        - ## 定位、修改策略对应的注册表
          collapsed:: true
          - 聊完了组策略，再来看看注册表中的策略。它俩看似风马牛不相及，实际上却是「貌离神合」。
          - 不过需要注意的是，相较于操作管理模板，修改注册表的风险会更大一些。建议读者在实际上手之前按照本文「策略的备份和迁移」章节的步骤先导出相关的注册表项以备不测。实在没有把握的话，也可以手动创建系统还原点（搜索「创建还原点」即可），这会执行注册表快照，便于在系统出错时回滚。
          - ### 管理模板和注册表的关系
            collapsed:: true
            - 简单思考便不难发现：组策略编辑器并不是执行应用策略的唯一入口。因为默认情况下 Windows 家庭版无法使用组策略，但恶意软件仍然可以借助策略锁定浏览器主页；目前市面上各种 Windows 配置软件，如 [Dism++](https://sspai.com/link?target=https%3A%2F%2Fgithub.com%2FChuyu-Team%2FDism-Multi-language%2Freleases%2F)、[Shutup10++](https://sspai.com/link?target=https%3A%2F%2Fwww.oo-software.com%2Fen%2Fshutup10)，并不限制 Windows 版本……注册表是唯一可能的解释。
            - 实际上，管理模板中的项目和注册表往往存在着对应关系，大部分情况下，配置管理模板就是在修改注册表。
            - 根据微软的组策略 [API 文档](https://sspai.com/link?target=https%3A%2F%2Flearn.microsoft.com%2Fen-us%2Fprevious-vesions%2Fwindows%2Fdesktop%2Fpolicy%2Foptions-for-providing-policy)，要为自己的应用提供策略有三种方式：
              - * 继承系统策略；
              - * 编写组策略扩展；
              - * 用管理模板文件拓展**基于注册表**的策略。
            - 第三种正是本文讨论的主题。管理模板文件我们在上一节中已经下载、安装过，之前说过，`admx` 其实是 XML 文件，以下是删减过的 Edge 控制边栏的管理模板内容：
            - ```
               <policy class="Both" …… key="Software\Policies\Microsoft\Edge" name="HubsSidebarEnabled" …… valueName="HubsSidebarEnabled">
                    <parentCategory ref="microsoft_edge"/>
                    <supportedOn ref="SUPPORTED_WIN7_V99"/>
                    <enabledValue>
                      <decimal value="1"/>
                    </enabledValue>
                    <disabledValue>
                      <decimal value="0"/>
                    </disabledValue>
              </policy>
              ```
            - `key` 定义了注册表的位置，`name` 指定注册表键的名称，`<decimal value="1"/>` 定义了注册表值的类型和数据。在管理模板中配置该策略后，你一定能在注册表找到这个键值：
            -
            - 所以说，即便没有组策略编辑器，只要我们能知道该添加、修改具体哪些注册表项目，仍然可以配置策略。
          - ### 定位注册表策略的位置
            collapsed:: true
            - 根据微软的 [设计](https://sspai.com/link?target=https%3A%2F%2Flearn.microsoft.com%2Fen-us%2Fprevious-versions%2Fwindows%2Fdesktop%2Fpolicy%2Fimplementing-registry-based-policy)，和策略有关的注册表一般位于以下两个位置3：
            - ```
              \Software\Policies\
              \Software\Microsoft\Windows\CurrentVersion\Policies\
              ```
            - 和组策略编辑一样，注册表中同样分为计算机配置和用户配置，因而对计算机（注册表简写 `HKLM`）生效的注册表策略在：
            - ```
              HKEY_LOCAL_MACHINE\Software\Policies\
              HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Policies\
              ```
            - 对当前用户（注册表简写 `HKCU`）生效的注册表策略在：
            - ```
              HKEY_CURRENT_USER\Software\Policies\
              HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Policies\
              ```
            - 根据同样的原则，除非有特殊需求，一般都建议直接修改 `HKLM` 下的注册表策略，计算机策略优先级高于用户策略。至于究竟要把键值放在哪一个位置，还应该根据应用开发者的意愿，但约定成俗的原则是软件配置放在哪里，策略就放在哪里，具体表现便是：
            - Windows 紧密集成的各种组件一般放在 `\Software\Microsoft\Windows\CurrentVersion\Policies\` 下，比如文件资源管理器、Windows 语言模块、BitLocker；
            - 后来安装到系统的软件一般放在 `\Software\Policies\`，比如说 Edge、Chrome、Firefox；
            - 在这两个位置下，应用开发商一般用 `公司名\产品名\` 安排目录结构，例如 `Microsoft\Edge\`、`Mozilla\Firefox`。
          - ### 修改具体的键值
            collapsed:: true
            - 万事俱备，只欠一个具体的注册表键名和键值。
            - 其实查找的方案也非常简单：之前推荐过两个网站，在线搜索 Windows 默认管理模板的 [Group Policy Search](https://sspai.com/link?target=https%3A%2F%2Fgpsearch.azurewebsites.net%2F) 以及收集了众多第三方管理模板的 [Group Policy Home](https://sspai.com/link?target=https%3A%2F%2Fadmx.help%2F)。这两个网站除了提供策略的详细说明，同时也展示了每个策略对应的注册表键值：
            -
            - 当然，它们不是官方网站，更新可能有延迟。部分开发商在其官网文档里也给了修改注册表的指导。例如在 Edge、Chrome 等 Chromium 系浏览器地址栏输入 `about:policy`, 在 Firefox 地址栏输入 `about:policies` 能得到已生效的策略及所有策略的详细信息：
            - 即便以上两种方案都找不到想要的结果，我们也可以返璞归真：像本节一开头展示的那样读 `.admx` 文件，找到放在模板中的 `key`、`name` 和数据。
            - 举个例子，我想用注册表的方式禁用 Firefox 在后台自动更新。那么我大概率应该在下面的注册表位置创建具体的键值：
            - ```
              HKEY_LOCAL_MACHINE\Software\Policies\Mozilla\Firefox\
              ```
            - 根据 Firefox 内置文档以及[admx.help](https://sspai.com/link?target=https%3A%2F%2Fadmx.help%2F%3FCategory%3DFirefox%26Policy%3DMozilla.Policies.Firefox%3A%3ADisableAppUpdate)的说法，我需要搜索「注册表」或者 `regedit.exe` 打开注册表编辑器，在上面的位置创建名为 `DisableAppUpdate` 的 REG_DWORD 键，将其值设为 `1`，然后重启 Firefox，策略就生效了。
          - ### 第三方软件
            collapsed:: true
            - 前面已经反复提过，许多系统设置自定义应用本质上是修改了注册表策略。不想手动做这些工作的话，我个人推荐使用 [Shutup10++](https://sspai.com/link?target=https%3A%2F%2Fwww.oo-software.com%2Fen%2Fshutup10)：
            - 此软件同样提供了策略的导入导出功能，并且给每一个选项都提供了评级和说明。
            - 但这类软件的问题也很明显，由于不可能为了个别策略单独发版，它们无法解决一些新近出现的问题。
            - 另外，所有注册表中的更改并不会反映在管理模板中。例如，我在上一小节中修改了 Firefox 的更新设置，此时打开管理模板，Firefox 对应策略仍然是未配置状态。倘若我又通过管理模板将后台更新打开，那么注册表项会跟着更新。这种现象在用户自行修改时并不会造成困扰，但第三方软件也会更改注册表中的某些策略，这个时候我们是无法通过管理模板发现的。
        - ## 策略的备份和迁移
          collapsed:: true
          - 花了许多心力配置的各种策略，如果在迁移环境后要重新一个个配置一遍，着实有点说不过去。而组策略管理模板及注册表的好处就在于可以方便得备份和迁移。
          - 顺着上一节注册表的话头继续说下去，应用策略必然存储在上面列举的四个位置中，而 Windows 注册表支持导入导出，因而，只要定位到这些位置，右键导出对应注册表便能完成备份：
          - 迁移到新环境后，双击该注册表文件就能完成策略的导入。
          - 组策略管理模板同样支持导入导出，下面两个目录存储着组策略管理模板的配置数据，如果目录为空，说明没有做任何配置，忽略即可：
          - ```
            C:\Windows\System32\GroupPolicy
            C:\Windows\System32\GroupPolicyUsers
            ```
          - 定位到这两个目录，复制其子目录下的 `Registry.pol` 文件，将所有的 `Registry.pol` 粘贴到新电脑对应的位置中，此时打开组策略编辑器应该能看到所有配置已同步。
          - 不过这仅仅是表层上的同步，策略并没有真正生效。复制好 `Registry.pol` 后，在 shell 中执行以下命令一次才能真正完成迁移，部分策略可能需要重启才能生效：
          - ```
            gpupdate /force
            ```
          - 如果你觉得这样复制粘贴有些麻烦，那么可以自行将它们包装成脚本。亦或者使用包装好的第三方工具[LGPO.exe](https://sspai.com/link?target=https%3A%2F%2Flearn.microsoft.com%2Fen-us%2Farchive%2Fblogs%2Fsecguide%2Flgpo-exe-local-group-policy-object-utility-v1-0)。
          - 此工具能够解析 `Registry.pol` 文件以及一键备份、应用管理模板，使用方式如下：
          - ```
            lgpo /b <path> # 备份
            lgpo /g <path> # 从备份恢复
            ```
          - 除了可以用来备份迁移，变通使用以上几种方案也适用于批量删除、还原策略。删除特定的注册表可以撤销策略；删除 `Registry.pol` 然后执行 `gpupdate /force` 能重置所有组策略管理模板。
        - ## 总结
          collapsed:: true
          - 无论你用了多长时间的 Windows，这个系统总有一些功能是你从未仔细探索，亦或没有勇气去探索的，本文的主角——本地组策略编辑器和注册表策略——就是其中一例。
          - 虽然它们已经存在多年，但可以肯定，不管是 Web 几点零，也不管是图文时代、短视频时代还是元宇宙时代，只要微软不修改这些底层的系统结构，那么组策略、注册表仍将是 Windows 用户手上的利器。
